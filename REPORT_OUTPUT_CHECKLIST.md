# 静的レポート出力チェックリスト

田中祐一AIの添削モードで、静的HTML/MDレポートを公開する前に必ず通す検収リストです。
「生成したつもり」「コミットしたつもり」ではなく、公開URLで見える状態まで確認します。

## 毎回の必須ゲート

### 1. 仕様・参照元の確認
- [ ] ユーザーの最新指示を読み直し、今回の出力範囲を1文で定義した
- [ ] 参照すべき過去版・参考URL・スクリーンショットを確認した
- [ ] 期待する成果物が「ビジュアルレポート」「テキストレポート」「素材集」の3点セットか確認した
- [ ] 増田 W3EV 5日チャレの現在の合格形と見比べ、同じ構造を再現できているか確認した
- [ ] 既存版への上書きか、新規URL作成かを確認した
- [ ] `REPORT_OUTPUT_ARCHITECTURE.md` のページ構造・リンク生成ルールと矛盾していない
- [ ] 素材ID、ページ種別、URLを一覧化し、手打ちリンクを増やしていない
- [ ] アーキテクチャ表にない `materialId` を導入していない
- [ ] 仕様変更時に `REPORT_OUTPUT_ARCHITECTURE.md`、`REPORT_OUTPUT_CHECKLIST.md`、`scripts/check-report-output.mjs` を同時に更新した

### 2. 3点セットの存在
- [ ] `index.html` が存在し、ファネルレポートポータルとして3種類の入口を表示している
- [ ] `visual-report.html` が存在する
- [ ] `text-report.html` が存在する
- [ ] `visual/materials.html` が存在する
- [ ] 3点がお互いにリンクしている
- [ ] 3点の公開URLがすべてHTTP 200で開く
- [ ] 「3点セット」という内部呼称を顧客向け見出しに使っていない
- [ ] サイドバーに表示した素材IDすべてに対して、対応するビジュアルページが存在する
- [ ] テキスト分割仕様の場合、サイドバーに表示した素材IDすべてに対して、対応するテキストページが存在する

### 3. ビジュアルレポート
- [ ] 全ページに左サイドバーが固定表示されている
- [ ] 左サイドバーはファネル素材の1層目一覧を表示し、右側フィードバック項目を重複表示していない
- [ ] サイドバーの素材リンクはビジュアルページ内では `visual/<materialId>.html` に遷移する
- [ ] LP・サンクス・個別説明会LPなどのページ素材は、顧客向けに「第1層/第2層/第3層」と表示していない
- [ ] LP系ページは、左側にキャプチャー、右側に番号付きフィードバックが並ぶ構成になっている
- [ ] 指摘番号と画面上の該当箇所が対応している
- [ ] 指摘一覧は必要に応じてアコーディオンで閉じられる
- [ ] 右側フィードバックは「高・中・小」の優先度で分類され、閉じた状態でも優先度が見える
- [ ] 全体ページに、素材別の指摘件数と優先度（高・中・小）内訳が表示されている
- [ ] メール・LINE・動画など時系列素材は、時系列で追える構成になっている
- [ ] 動画ページは冒頭キャプチャー、書き起こし該当箇所、映り方評価、本文評価が分離されている
- [ ] 良い点も書かれており、悪い点だけの指摘になっていない
- [ ] 「関連レポート一覧」など重複・不要な導線が残っていない
- [ ] `テキストレポート対応` や `テキストレポート 全体感の1` のような内部対応メモが表示されていない

### 4. テキストレポート
- [ ] サイドバーが存在する
- [ ] サイドバーから本文目次、または素材別テキストページへ移動できる
- [ ] サイドバーからビジュアルレポートと素材集へ移動できる
- [ ] テキストレポートの素材リンクが `visual/<materialId>.html` に誤遷移しない
- [ ] 素材別テキストページがある場合、サイドバーの素材リンクは `text/<materialId>.html` に遷移する
- [ ] 暫定的に1枚HTMLの場合でも、顧客向けHTMLではなく検収メモまたは完了報告に理由と解消予定が明記されている
- [ ] 冒頭にファネルレポートポータル、ビジュアルレポート、原本素材集への導線がある
- [ ] 本文が「用語整理 → 素材対応表 → ファネル全体像 → 全体感 → 指摘件数と優先度内訳 → 個別フィードバック」の順で読める
- [ ] 個別フィードバックやフェーズ表に「高・中・小」の優先度が入り、「大」「低」などの旧表記が残っていない
- [ ] 本文の見出しにアンカーIDが付いている
- [ ] マーケター向けの白基調デザインで、右側本文は1カラムで読める

### 4.5 内容品質・田中基準との照合
- [ ] 各素材の指摘が3件前後で終わっている場合、素材量に対して薄すぎないか再読した
- [ ] 田中祐一の過去Loom/音声添削・過去の合格版レポートと見比べ、指摘の粒度・踏み込み・言い換え案の量が近いか確認した
- [ ] 「現状」「問題点・考察」「解決策案」がそれぞれ別の情報になっており、同じことの言い換えで水増しされていない
- [ ] ファネル全体、各素材、横断論点の3階層で指摘が存在する
- [ ] 高優先度の指摘には、成約率・説明会着席率・離脱防止など、なぜ優先度が高いかの理由が書かれている

### 5. 素材集
- [ ] 素材集は分析レポートではなく、原本素材台帳として表示されている
- [ ] 素材集にキャプチャー評価、良い点、改善点などの評価コンテンツが混ざっていない
- [ ] 素材集に「良い点」「改善案」「優先度 高/中/小」「指摘」などの評価語が本文コンテンツとして混ざっていない
- [ ] 実際のURL、保存済み原本MD、内容種別が対応表で確認できる
- [ ] LP・サンクス・個別説明会LPなどの本文MDが見られる
- [ ] 動画URLまたは埋め込みと書き起こしMDがセットで見られる
- [ ] 動画書き起こしに10分ごとのタイムスタンプがある
- [ ] ステップメール/LINEは一覧から1段深く入り、通数・日付・本文が確認できる
- [ ] 1段深いページから「元ファネルの一覧へ」戻れる
- [ ] 画面上部で取得済み素材の全体件数とカテゴリ別件数が分かる
- [ ] 画面が実際に参照しているMDリンクを使って、公開URLでサンプルHTTP 200を確認した

### 6. 表記・ブランディング
- [ ] 「田中祐一AI」に統一されている
- [ ] 「田中祐一OS」「田中祐一ーOS」「ナレッジOS」「TanakaKnowledgeOS」など旧表示が残っていない
- [ ] HTML/JS/CSSだけでなく、公開される原本MD・TXTにも旧表示が残っていない
- [ ] 顧客向け画面に内部検討用の階層名・作業メモが露出していない
- [ ] 顧客向けHTML全体に「3点セット」「第1層」「第2層」「第3層」が残っていない

### 7. 公開反映確認
- [ ] ローカルファイルだけでなく、GitHub Pages/Vercelなど本番URLで確認した
- [ ] キャッシュバスター付きURLでも最新表示を確認した
- [ ] 404、client-side exception、別ページへの誤リンクがない
- [ ] 修正前に問題だった旧表示が消えていることを確認した
- [ ] `--public-base` 付きで自動チェックを実行しPASSした

### 8. コミット・報告
- [ ] `git status` で意図した変更だけが出ている
- [ ] 検査コマンドを実行してPASSした
- [ ] コミット・プッシュした
- [ ] 完了報告に、コミットID、確認した本番URL、チェック結果、残リスクを含めた

## 自動チェック

基本コマンド:

```bash
node scripts/check-report-output.mjs masuda-5chere-ai-output-20260610 \
  --public-base https://tanakayuc.github.io/static-pages/masuda-5chere-ai-output-20260610
```

このコマンドで確認すること:

- 3点セットのファイル存在
- テキストレポートのサイドバー
- テキストレポートの優先度サマリー
- ビジュアルレポートの優先度別フィードバック分類
- 素材集の原本素材表示
- 旧名称の混入
- `visual/source-materials.js` とローカルMDリンクの整合
- 公開URL上の代表MDリンクのHTTP 200

## 今回の反省から固定する運用

- 手打ちURLではなく、生成HTML/JSが実際に使っているhrefを検査する
- リンク生成ルールとページ構造は `REPORT_OUTPUT_ARCHITECTURE.md` を正本とする
- 新規案件では、増田 W3EV 5日チャレの現在の合格形を基準テンプレートとして構造差分を確認する
- 素材IDからビジュアル、テキスト、原本素材のURLを生成し、同じリンクを複数箇所に手入力しない
- 「ローカルではある」だけで完了にしない
- 「HTMLを開ける」だけでなく、画面上部にユーザーが探している情報が見えるか確認する
- ユーザーが参考URLを出した場合は、構造の差分を明示してから実装する
- 大きめの変更では、仕様管理・実装・QA・デプロイ確認の観点を分けて見る

## Definition of Done

完了とは、以下をすべて満たした状態です。

1. 期待された3点セットが公開URLで見える
2. ユーザーが指摘した旧表示・旧構造が消えている
3. 自動チェックがPASSしている
4. 目視で主要画面を確認している
5. コミット・プッシュ済みで、報告に証跡がある

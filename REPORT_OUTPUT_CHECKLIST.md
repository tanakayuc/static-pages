# 静的レポート出力チェックリスト

田中祐一AIの添削モードで、静的HTML/MDレポートを公開する前に必ず通す検収リストです。
「生成したつもり」「コミットしたつもり」ではなく、公開URLで見える状態まで確認します。

## 毎回の必須ゲート

### 1. 仕様・参照元の確認
- [ ] ユーザーの最新指示を読み直し、今回の出力範囲を1文で定義した
- [ ] 参照すべき過去版・参考URL・スクリーンショットを確認した
- [ ] 期待する成果物が「ビジュアルレポート」「テキストレポート」「素材集」の3点セットか確認した
- [ ] 既存版への上書きか、新規URL作成かを確認した

### 2. 3点セットの存在
- [ ] `visual-report.html` が存在する
- [ ] `text-report.html` が存在する
- [ ] `visual/materials.html` が存在する
- [ ] 3点がお互いにリンクしている
- [ ] 3点の公開URLがすべてHTTP 200で開く

### 3. ビジュアルレポート
- [ ] LP・サンクス・個別説明会LPなどのページ素材は、顧客向けに「第1層/第2層/第3層」と表示していない
- [ ] LP系ページは、左側にキャプチャー、右側に番号付きフィードバックが並ぶ構成になっている
- [ ] 指摘番号と画面上の該当箇所が対応している
- [ ] 指摘一覧は必要に応じてアコーディオンで閉じられる
- [ ] メール・LINE・動画など時系列素材は、時系列で追える構成になっている
- [ ] 「関連レポート一覧」など重複・不要な導線が残っていない

### 4. テキストレポート
- [ ] サイドバーが存在する
- [ ] サイドバーから本文目次へ移動できる
- [ ] サイドバーからビジュアルレポートと素材集へ移動できる
- [ ] 「これが今回のレポートです」として3点セットのURLが明示されている
- [ ] 本文の見出しにアンカーIDが付いている

### 5. 素材集
- [ ] 素材集は分析レポートではなく、原本素材台帳として表示されている
- [ ] 実際のURL、保存済み原本MD、内容種別が対応表で確認できる
- [ ] LP・サンクス・個別説明会LPなどの本文MDが見られる
- [ ] 動画URLと書き起こしMDがセットで見られる
- [ ] ステップメール/LINEは通数・日付・本文が確認できる
- [ ] 画面上部で取得済み素材の全体件数とカテゴリ別件数が分かる
- [ ] 画面が実際に参照しているMDリンクを使って、公開URLでサンプルHTTP 200を確認した

### 6. 表記・ブランディング
- [ ] 「田中祐一AI」に統一されている
- [ ] 「田中祐一OS」「田中祐一ーOS」「ナレッジOS」「TanakaKnowledgeOS」など旧表示が残っていない
- [ ] 顧客向け画面に内部検討用の階層名・作業メモが露出していない

### 7. 公開反映確認
- [ ] ローカルファイルだけでなく、GitHub Pages/Vercelなど本番URLで確認した
- [ ] キャッシュバスター付きURLでも最新表示を確認した
- [ ] 404、client-side exception、別ページへの誤リンクがない
- [ ] 修正前に問題だった旧表示が消えていることを確認した

### 8. コミット・報告
- [ ] `git status` で意図した変更だけが出ている
- [ ] 検査コマンドを実行してPASSした
- [ ] コミット・プッシュした
- [ ] 完了報告に、コミットID、確認した本番URL、チェック結果、残リスクを含めた

## 自動チェック

基本コマンド:

```bash
node scripts/check-report-output.mjs masuda-5chere-ai-output-20260610 \
  --public-base https://tanakayuc.github.io/static-pages/masuda-5chere-ai-output-20260610
```

このコマンドで確認すること:

- 3点セットのファイル存在
- テキストレポートのサイドバー
- 素材集の原本素材表示
- 旧名称の混入
- `visual/source-materials.js` とローカルMDリンクの整合
- 公開URL上の代表MDリンクのHTTP 200

## 今回の反省から固定する運用

- 手打ちURLではなく、生成HTML/JSが実際に使っているhrefを検査する
- 「ローカルではある」だけで完了にしない
- 「HTMLを開ける」だけでなく、画面上部にユーザーが探している情報が見えるか確認する
- ユーザーが参考URLを出した場合は、構造の差分を明示してから実装する
- 大きめの変更では、仕様管理・実装・QA・デプロイ確認の観点を分けて見る

## Definition of Done

完了とは、以下をすべて満たした状態です。

1. 期待された3点セットが公開URLで見える
2. ユーザーが指摘した旧表示・旧構造が消えている
3. 自動チェックがPASSしている
4. 目視で主要画面を確認している
5. コミット・プッシュ済みで、報告に証跡がある

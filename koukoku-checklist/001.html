広告運用チェックリスト（社内ルール）

最終更新：2026-05-11（最新の議事録日付）

このチェックリストは広告ミーティング（田中なんでもMTG）の議事録から抽出した運用ルールです。
議事録更新のたびにあわせて更新します。
参照元：2026年3月30日・4月6日・4月13日・4月20日・5月11日の各議事録

1. アカウント管理

Metaアカウントの構造ルール

  • [ ] Metaのアカウント構造を正しく理解して管理する：ビジネスマネージャー（BIZマネ）が最上位。その下にFacebookページ・インスタアカウント・広告アカウント・ピクセルが並列で紐付く（階層ではなく放射状）
  • [ ] 個人アカウント1つにつきビジネスマネージャーは2つまでとする
  • [ ] 新規ビジネスマネージャーを作成した場合、最初の広告アカウントは1つのみ作成可能であることを前提に計画を立てる（運用実績が積まれると上限が増える）
  • [ ] 予算枠（Spending Limit）はビジネスマネージャー単位で管理されることを把握する
  • [ ] Facebookページとインスタアカウントは1対1で紐付ける。最近は紐付けなしだと広告が出せないケースが増えているため、両方必ず準備する
  • [ ] クライアント案件の広告運用は「クライアント自身のビジネスマネージャーに広告運用者として招待してもらう」形を基本方針とする。自社アカウントで代行する形は審査・インスタ紐付け・予算枠の問題で限界が来るため採用しない

新規案件立ち上げ時のセットアップ

  • [ ] 新規案件立ち上げ時は初回MTGでクライアントと同席のうえ、以下を一緒に完了させる：
  • ビジネスマネージャーの作成
  • Facebookページの作成
  • インスタアカウントの紐付け（インスタパスワードが必要なため本人作業）
  • 広告運用者（ザ・リードメンバー）をビジネスマネージャーへ招待
  • [ ] 新規ビジネスマネージャーは予算上限が低い状態からスタートするため、最初は月30万円規模のやさしい案件から始めて信頼を積む
  • [ ] 新規アカウントで立ち上げる場合、拡散広告（インプ稼ぎ）よりシンプルなCV広告を細々と流して地ならしする

アカウントBANへの対処

  • [ ] 個人アカウントがBANされた場合：セルフィー動画（右・上・左と顔を動かす本人証明）→パスポート提出の手順で対応する
  • [ ] 個人アカウントにペナルティが付いた状態でも、別管理者を追加すれば広告運用は継続可能
  • [ ] 安全性を優先する場合は新規個人アカウント＋新規BIZマネで立て直す。広告アカウントを新規作成するとキャンペーン履歴は消えるが、ピクセルは引き継げる
  • [ ] ビジネスマネージャーの広告アカウントが上限（10個）に達したら新規作成不可となるため、使っていないアカウントの棚卸しを定期的に行う
  • [ ] アクセストークンが崩れる問題が発生した場合は、個人のFacebookセッションに紐づくトークンではなく「システムユーザートークン」に切り替える

APIトークン管理

  • [ ] Meta広告のAPIはシステムユーザートークンを使用する（個人のFacebookセッションに紐づくトークンはログアウトで崩れるため使わない）
  • [ ] システムユーザートークンは定期的に有効性を確認し、崩れていたら田中に直接報告する

2. ターゲティング・オーディエンス設定

Metaアルゴリズムの前提理解

  • [ ] Metaは今後「詳細ターゲット廃止・ブロード配信化」に向かっていることを前提に設計する（2026年4月時点で共有済み）
  • [ ] ブロード配信下では、オーディエンス設定に頼らずクリエイティブのテキスト・画像からAIが配信先を判断する。広告文・バナーの中に「どんな人へ届けたいか」を示すワードを明示的に入れる
  • [ ] Metaのアルゴリズムは広告文の内容をもとにターゲットを自動決定するため、競合と同じメッセージを出すと同じターゲットに同じ広告が並ぶ状態になる

配信先・オーディエンスの選定

  • [ ] グローバル配信（日本以外）は、CPMが安くても実際の成約には繋がらないケースが多いため、日本限定配信を基本とする（波動占い案件の事例より）
  • [ ] 類似オーディエンスを活用する場合、CPAが悪化したら即座に過去最もCPAが安かったオーディエンスに切り替えて再検証する
  • [ ] 既存受講生は除外リストとして必ず設定する（例：次世代型セールスの既存45分受講者を除外）
  • [ ] 「今すぐ客」（すぐに商品が欲しい・必要な人）向けと「潜在層」向けを別軸で並走させ、LTV・CPOの取り方を変える

予算帯別の広告セット設計

  • [ ] 一日10,000円前後の予算帯では、広告セットへの登録クリエイティブ数は最大3つ程度とする
  • [ ] 大予算アカウント（一日100万円以上）とは運用の考え方が異なるため、同じルールを適用しない

3. クリエイティブ制作

制作の基本方針

  • [ ] クリエイティブには制作時の「仮説」と「ターゲティング設定」を必ずスプレッドシートまたはPalmoに記録する（数値だけでなく意図と結果の文脈を残す）
  • [ ] 「インスタで反応が良いもの」と「広告で成約に繋がるもの」は必ずしも一致しないことを前提に判断する
  • [ ] コミュニティ内で刺さった内容をそのまま広告に持っていくのは危険。広告は手法・ノウハウ知りたい層から入りライブで教育される流れを踏まえる

訴求の作り方

  • [ ] 競合と同じ横並びの訴求を避ける。競合が使っていない切り口・角度で出すことでCPMを下げられる
  • [ ] 「今まで誰も言っていなかったメッセージ」を探す。特に、ターゲットが他社で感じている不満・失敗体験を主軸に据えると差別化になる
  • [ ] 「見飽きられた」と「刈り取り切った」は別の事象として判断する。同じターゲットに新しい訴求を投入し続けることで記憶に残り、複数回広告を見た後で申し込む流れが生まれる
  • [ ] 「競合潰し系」広告（例：「英語教室に通わせると失敗します」）は、バックエンド成約率が上がりやすい。権威性広告を継続しながら並走させる
  • [ ] データと具体的なメカニズムを組み合わせた訴求が有効（例：「バイリンガルの成功率7%…93%の側に入ってしまう」）
  • [ ] 広告で差別化できた状態で入ってくる見込み客は熱量が高い。「まずは情報収集」温度で入った場合は最後の押しで競合に負ける構造になりがち
  • [ ] 問題認識の低い層を引き上げる教育コストが高い広告ばかりにならないようバランスをとり、「今すぐ客」を狙う広告も加える

クリエイティブ素材の種類と使い分け

  • [ ] 静止画・動画・カルーセルを混在させ、一種類に集約しすぎない（いざ反応が落ちた時に切り替えられるよう備える）
  • [ ] 競合が写真ばかり使っている場合はイラストで差別化する。イラストはAI感が出にくいためコピーの訴求で戦いやすい
  • [ ] AI生成の写真はユーザーがAIと気づく傾向があるため、写真を使う場合は「インパクトある一枚」が必要
  • [ ] 動画クリエイティブはサムネイル・先頭数秒のコピーが評価の中心になる。画像と動画で見るべき指標が異なることを意識する
  • [ ] 「セッション切り抜き」「リアルな現場映像」は作り込みクリエイティブより成約に繋がりやすい傾向がある
  • [ ] セッション動画は被写体（お客様）の顔が映らない構図で撮影することで同意取得の負担を減らせる（海外でも講師背中・お客の後ろ姿の構図が多い）
  • [ ] カルーセル広告は一枚目でターゲットに刺さる悩みを出す。二枚目以降は情報を詰め込みすぎず見切れが出ないデザインにする

クリエイティブリサーチ手法

  • [ ] YouTubeで登録者数に対して再生が異常に高い動画（vidIQ等でリサーチ）のサムネイル文言を広告コピーに転用する。YouTubeは能動的クリックの媒体のため、伸びている動画は需要が確実にある証拠
  • [ ] YouTubeリサーチの条件目安：登録者500〜3万人（有名すぎを除く）、再生倍率が高いもの
  • [ ] ストーリー広告（蕎麦屋型など）は集客しやすいが、バックエンドに繋がりやすい客を連れてくるかどうかは別問題。続きをLPで読ませる構造にし、広告内で見せすぎてLP不要にしない
  • [ ] お客様の声をLP冒頭に並べる（アイコン・後ろ姿でも可）だけで差別化になるケースがある

AI生成ツールの活用

  • [ ] ChatGPT Image（現在Image 2）：文字化けが解消され高品質な広告画像がプロンプト一発で生成可能。AI感が出にくく今すぐ使えるタイミング
  • [ ] メタAIを広告審査の事前相談に活用する（審査通過の可否を事前確認できる）
  • [ ] 「感覚を数字で言語化させる」：自分が言葉にできないもやもやをAIに補足させながら進めると精度が上がる（例：「縦が長い」→「934ピクセルにしてほしい」）

クリエイティブの審査・NGルール

  • [ ] 広告審査が通りにくい業界（スピリチュアル・医療隣接等）は、本のLPを使って広告を通す方法を検討する（「広告が通らない業界で本のLPを使う」戦略）
  • [ ] 広告文・バナーのNGワードについては業界ごとに確認し、メタAIで事前審査を行う

4. 予算管理

初期予算と目標CPAの基準

  • [ ] 目標CPAを超えたら即停止して新しい訴求を出す。粘りすぎない
  • [ ] 一日の予算帯（1万円前後）に合った運用規模（広告セット数・クリエイティブ数）で設計する
  • [ ] 新規ビジネスマネージャーの最初は月30万円規模のやさしい案件から始め、信頼（予算枠）を積み上げてからスケールアップする

スケールアップの判断基準

  • [ ] CPAが安定している権威性広告を続けながら、新しい訴求を並走させる形でスケールする
  • [ ] CTRが高ければCPMが下がり、低ければ上がる傾向が顕著にある。CPMが高い場合はCTR改善（訴求変更）で対処する
  • [ ] ROASの目安：250%以上を目標とし、200%程度は改善余地ありと判断する（自社広告の過去実績より）

代理店経由と自社運用の比較

  • [ ] 代理店経由の広告ROAS目安：250%程度（2026年3月時点の田中の認識）
  • [ ] 自社直接運用でもROAS200%程度は確保できているが「良くはない」水準として認識する

5. 運用フロー・担当連携

広告ナレッジの格納場所

  • [ ] 広告ノウハウ・議事録・クリエイティブはすべて 42_広告ノウハウ（リポジトリ直下）に集約する
  • [ ] 売れたクリエイティブも同フォルダに格納する（山岸・担当者制作のものを含む）
  • [ ] クリエイティブはGitHubにURLで履歴化する。リポジトリが重くなる場合はGoogleドライブも検討する
  • [ ] チャットでのクリエイティブ管理は行わない。フォルダ内でフィードバックと履歴を残す

レポート・数値報告

  • [ ] 毎週月曜朝9時に広告レポートを自動出力し、担当者が考察を加えて再出力する（Palmoで実装予定）
  • [ ] CVR計算はMeta管理画面の数値（CV÷インプレッション）ではなく、正しい計算式（CV÷クリック数）で行う
  • [ ] UTAGEのAPI連携とMetaのexclusion設定がずれるとCV数値と実数に乖離が生じる。管理画面数値と実数（スプレッドシート）の二系統を確認する
  • [ ] LP別・クール別の数字と仮説をセットで追跡し、成約率・参加率との相関を記録する

パラメータ管理

  • [ ] 広告パラメータは担当者間で共通ルール（CID命名規則）を使う。各自の方言を使わない
  • [ ] パラメータの精度はブラウザのセキュリティ設定等により100%ではない（大半は追跡できているが例外がある）ことを前提に分析する
  • [ ] 複数回広告登録がある場合の紐付けは「最後のタッチポイントを採用」する

チーム内連携

  • [ ] 広告担当は「待ち」ではなく能動的に動く。「この日までにこれが必要だが進捗はどうか」と先方・社内メンバーにぐいぐい確認する
  • [ ] セールス担当からの客層・お断り理由などの情報を広告担当が定期的に収集し、クリエイティブ改善に活かす
  • [ ] クリエイティブごとに仮説・フィードバック履歴をPalmoに入力する（数値だけでなく意図と結果の文脈を残す）
  • [ ] 広告担当がバックエンドの成約状況をリアルタイムで確認できる状態を維持する

6. 注意事項・過去の失敗事例

クリエイティブ関連の失敗

  • MetaのAI最適化でLP3寄りに偏った結果、CVRはLP3が良く見えたが「参加率」が悪く成約も落ちた。管理画面の最適化に任せすぎず、LP別の参加率・成約率を独自に追跡する（おうちバイリンガル26クールの事例）
  • カルーセルは一枚目でターゲットに刺さらないと全体のCPAが悪化する。デザインの詰め込みすぎで見切れが出るとCTRが下がる
  • 動画広告は全体的にCPMが高くなりやすい（7,000〜10,000円超）。権威性クリエイティブだけがCPM2,000円台で安定している事例あり
  • ストーリー広告で内容を見せすぎると「続きをLPで見る動機」がなくなりCVが低下する（老舗メガネ屋ストーリーの事例）
  • グローバル配信はCPMが安くても日本以外では成約に繋がらなかった。配信地域は日本に絞る（波動占いの事例）

アカウント・システム関連の失敗

  • 個人のFacebookセッションに紐づくアクセストークンはログアウトで崩れる。システムユーザートークンに全件切り替えが必要
  • 担当者ごとにパラメータルールがバラバラだったため、データ統合時に「方言が混在」した状態が発覚。パラメータルールは最初から統一する（2026年4月13日MTGで発覚）

ターゲティング・オーディエンス関連

  • 「LP2 vs LP3」でMetaがLP3に偏った配分を自動決定したが、実際の成約率はLP2が2倍以上高かった。MetaのAI配分に完全に任せず、自社指標（説明会参加率・成約率）で独自に評価する
  • 既存の選択肢に満足していない層（他社への不満を持つ層）への訴求が強い。「お断り理由」「他社比較」の情報をセールス担当から積極的に集め、訴求に反映する

訴求と商品内容のずれ

  • 広告訴求とチャレンジ・セッション内容が整合していないと成約率が下がる。「クリエイティブで来た客が期待していたものとサービス内容がずれる」状態を作らない
  • SNSで刺さる内容≠広告で成約に繋がる内容。インスタの反応を見てそのまま広告に転用しない

付録：Palmo（広告管理システム）の使い方

Palmo は社内広告管理システム（田中開発）。以下の機能を提供する：

  • 広告クリエイティブの一覧・フィルター表示
  • クリエイティブごとの仮説・考察・フィードバック履歴の入力
  • Meta APIとの連携による数値の自動取得
  • UTAGEのWebhookデータの取り込み
  • 毎週月曜9時の自動レポート出力

バグ報告の手順：
- UIバグ → Palmo内のバグ報告機能（AIが自動修正）
- 接続系の問題 → 田中に直接報告

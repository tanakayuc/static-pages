AI社員化プロジェクト 設計方針まとめ

このドキュメントは、2026年4月13日のAI構築に関する設計ディスカッションをまとめたものです。
チームメンバーへの共有と、今後のエージェント構築の指針として使ってください。

結論：少数から始めて1体ずつ確実に増やす

世界の先行事例（Gartner・Microsoft調査）では、最初から多数のエージェントを並べた組織の40%以上が2年以内に失敗しています。成功しているのは「1体ずつ確実に機能させてから増やす」組織です。

エージェントを増やすことは「人を増やす」のとは違い、エージェント間の調整コストが非線形に増大します。5体のエージェントは、1体と比べて25倍の管理コストになりうると言われています。

増やす判断基準はこの3つだけです：

  1. 「特化」が必要なとき（この業務には専門知識が要る）
  2. 「並列処理」が必要なとき（同時に複数タスクをこなす必要がある）
  3. 「セキュリティ境界」が必要なとき（データを分離しなければならない）

この3つのどれにも当てはまらなければ、1体のエージェントを強化する方が正解です。

スキルとエージェントの違い

混同しやすいので整理します。

スキル エージェント
何か やり方を書いたカード（手順書） 役割を持った担当者
誰が動かすか 人間が「やって」と言う 自分で判断して動く
記憶 ない（毎回リセット） ある（文脈を引き継げる）
トリガー 手動 イベント・スケジュール・手動すべて可

「スキルをエージェントに移す」というより「スキルをエージェントの中に組み込む」が正確な表現です。既存のスキルファイルは捨てるものではなく、エージェントの「作業手順」としてそのまま転用できます。

1エージェント・複数タスク・確認付きモデル

エージェントを業務の数だけ分けると、管理コストが爆発します。
正しい設計は「1体の中に複数タスクを定義し、最初は確認ゲートをつけて運用する」です。

これは「Supervised Autonomy（監督付き自律）」と呼ばれる業界標準のパターンです。

フェーズ 動き方
「見習い」 毎ステップ確認を求める
「半人前」 完了後に報告、差し戻し可
「一人前」 自動実行、異常時のみ通知
「熟練」 完全自動、ログだけ残す

同じタスクを「修正なし」で10回連続通過したら確認を外す。「修正あり」が連続2回来たらタスク定義を見直す。この数字基準で判断します。

フォルダ構造の設計

ナレッジ・社員仕様書・成果物の3つは性質が違う別物です。混在させると全部が腐ります。

knowledge/                  ← 唯一の真実（ここだけ更新する）
  LP添削/
    原則.md
    チェックリスト.md
    事例/
  セールス/
  プロモ設計/

agents/                     ← ナレッジを参照するだけ（コピーしない）
  LP添削担当/
    agent.md                ← knowledge/LP添削/ を参照
    tasks/
  ナレッジ管理担当/          ← 全体のバランスを見る
    agent.md

deliverables/               ← クライアント・受講生に渡す成果物テンプレ
  LP添削レポート/
  マニュアル/

各エージェントフォルダの中には以下を定義します：

agents/会議サポート担当/
  ├── agent.md          ← 役割・人格・基本方針（1エージェント）
  ├── task_文字起こし.md
  ├── task_議事録生成.md
  └── task_セールス分析.md

ナレッジ管理の鉄則：Single Source of Truth

ナレッジが分散・置いてけぼりになる根本原因は「ナレッジが複数箇所にコピーされていること」です。

解決策はシンプルです。ナレッジは1箇所にだけ書き、エージェント仕様書はそこを「指さす」だけにします。コピーしない。

「ナレッジ管理担当」エージェントを1体設置することで、以下が自動化できます：

  • 新しいLoom動画が処理されたとき、どの社員仕様書が影響を受けるか通知する
  • 各社員仕様書が参照しているナレッジのバージョンが最新か定期チェックする
  • ナレッジが更新されたのに社員仕様書が古いまま、という「ズレ」を検知して報告する

インプット・アウトプット・ナレッジの関係

添削業務を例に整理します。

層 内容 保存方針
インプット クライアントのLP草稿（XMind・Googleドキュメント） 案件フォルダに一時保存（後回しでいい）
アウトプット 田中さんのLoomフィードバック 文字起こし→knowledge/に格納
ナレッジ 「このパターンにはこう指摘する」という判断基準 knowledge/に集約・永続管理

Loom動画の文字起こしから始めるのが最速です。インプットデータの整理を先にしようとすると止まります。

ツールの使い分け

ツール 得意なこと
Cursor 画面を見ながら直接触る。人間が主役。仕様書を書く・育てる。
Claude Code 仕様書を読んで実行する。自動化・バッチ処理が主役。

社員の agent.md を書くのはCursorで人間が行い、その仕様書を読んで実際に動かすのはClaude Codeという分業が自然です。

エージェントのトリガー（起動方法）

現在のCursorでの「3時間に1回チェック」と同等の仕組みは、以下の方法で実現できます。

方法 PCが閉じていても動く ローカルファイルを触れる 備考
Claude Code スケジューラ（CronCreate） 〇 △ クラウド実行。API・GitHub連携向き
GitHub Actions 〇 × リポジトリ内のみ。すでに一部動いている
Mac launchd × 〇 PC起動中のみ。ローカルファイル処理向き

当面はMac launchdで十分動きます。完全自動化にはクラウド実行との組み合わせが必要ですが、それは後の話です。

チームへの役割分担

役割 担当内容
(a) 社員育成担当 業務知識をエージェントに移植する。agent.md を書く・育てる・テストする。
(b) 基盤構築担当 エージェントが使うツール・API・MCP連携を作る。
田中さん 全体構造の設計・クオリティチェック・手本エージェントの作成。

(a)は「AIに詳しい人」でなくてもできます。「その業務を一番よくわかっている人」が担うべきです。エージェントのコードは書けなくていい。agent.md に「こういうときはこうする」を日本語で書けるかどうかが基準です。

(b)が作ったMCP連携やAPIを、(a)が育てたエージェントが呼び出す構図になります。

スタッフへの課題設定

「AIを勉強してください」より「AI社員を1人育ててください」の方が機能します。

指示 問題点
AIを勉強してください 完了定義なし・成果物なし
ナレッジを溜めてください 完了定義なし・フィードバックなし
AI社員を1人育ててください 完了定義あり・動けば成功・動かなければ改善点が見える

スタッフへの具体的な指示例：

あなたが今担当している業務の中で、
「毎週必ず発生する作業」を1つ選んでください。

その作業を担当するAI社員を1人作ってください。
以下の3つを書けば完成です：

1. この社員は何をする人か（1〜2文）
2. 作業の手順（箇条書き）
3. うまくいったかどうかの判断基準

田中さんが先に「議事録作成の社員」「セールスフィードバック用の社員」の2体を手本として作り、「これを参考に各自1体作ってね」という形にすると、スタッフが一番動きやすくなります。

今急ぐべき優先順

  1. knowledge/ フォルダを新設して、LP添削のナレッジを1つ移す
  2. agents/ナレッジ管理担当/ を定義する（ナレッジと仕様書のズレを検知する役割）
  3. 手本エージェント（議事録・セールスフィードバック）を2体作る
  4. スタッフに「AI社員を1人育てる」課題を渡す

一気に整理しようとすると止まります。まずLoom動画を1本文字起こしして、3つのパターンを knowledge/ に書くことが最初のサイクルです。

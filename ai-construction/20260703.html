AI社員DB化・MCP移行設計と過去MD救済方針

作成日: 2026-07-03

結論

今すぐDB化の設計に入る判断は妥当。

ただし、いきなり「全データ参照MCP」を作るのではなく、既存のTanakaKnowledgeOSに「AI社員DB化モジュール」を足す形で進める。すでにTanakaKnowledgeOSにはSupabase、MCP、ナレッジ更新ループ、ナレッジ管理担当への引き渡しテーブルがあるため、別プロジェクトを新設すると認証、権限、ログ、ナレッジ候補、管理画面が二重化しやすい。

今回の方針は次の通り。

  1. Markdownは捨てない。人間が読む原本・成果物・ナレッジ正本として残す。
  2. Supabaseは、原本索引、評価履歴、分類履歴、利用ログ、AI社員稼働ログ、権限管理の台帳にする。
  3. MCPは、AI社員がDBを安全に参照・記録する入口にする。
  4. 過去2週間から1か月のMD仕分けは、旧評価軸による初期分類データとして取り込む。

過去MD作業は無駄ではない

今回つらく感じている原因は、過去の分類作業を「最終分類」として扱っていたことにある。DB化後は、分類を上書きせず「評価軸ごとの履歴」として重ねる。

つまり、今後は以下の考え方に変える。

これまで DB化後
基準が変わるたびにMDを再仕分けする 同じ原本に新しい評価軸の分類結果を追加する
どれが最新判断か分かりにくい rubric_versionで判断基準を明示する
以前の作業が消える legacy分類として残り、比較・再評価に使える
手作業の結果がファイル構造に埋もれる classification_runs / labelsとして検索できる

救済の要点は「過去MDを完成品として見る」のではなく、「旧評価軸で人間とAIが付けたラベル付きデータ」として見ること。

過去作業の救済方法

既存MDは、次の3層で取り込む。

層 内容 DB上の扱い
原本 Zoom、グルコン、個人コンサル、Chatwork、教材、LP、PDF、動画書き起こし source_assets
分割本文 見出し、話題、時系列、発言単位などのチャンク source_chunks
既存仕分け すでに付けた評価、分類、格納先、メモ classification_runs / classification_labels

過去の仕分けは legacy_md_2026Q2 のような評価軸バージョンとして保存する。これにより、新しい評価軸が出ても「過去作業を消してやり直す」のではなく、「同じ原本に新しい評価結果を追加する」運用になる。

最初に固定する4分類

田中さんが挙げた4つは、今後のナレッジOSの中核分類にする。

分類 意味 主な用途
判断基準・手順 再現性ある判断、手順、チェック項目 rules / skills / agent手順へ昇格
事例 実案件、実相談、成功例、適用例 RAG検索、セールス分析、教材化
失敗パターン 誤解、失敗、事故、うまくいかなかった型 QA、添削、リスク検知、教育
AI評価テスト AIの出力精度を測るための入力、期待回答、NG例 eval_cases / regression test

この4分類は排他的にしない。1つのチャンクが「事例」であり「失敗パターン」であり「AI評価テスト」になることもある。そのため、DBでは1チャンクに複数ラベルを付けられる構造にする。

4. AI社員台帳と稼働ログ

これにより、「AIを使いこなす社長」ではなく「AI社員が働き、評価され、育成される会社」に寄せられる。

5. 評価テスト

テーブル 役割

eval_runs 評価実行履歴
eval_results 各ケースの結果、スコア、差分、失敗理由

過去のグルコン・個人コンサルから良いケースを抜き出し、「田中祐一AIならこう答えてほしい」という期待出力を作る。これが将来の品質保証になる。

MCPの最小構成

最初から全機能をMCP化しない。まずはAI社員がDBを使うための最小ツールに絞る。

MCPツール 役割
search_source_assets 原本、チャンク、分類ラベルを横断検索する
get_source_asset 原本メタ情報、URL、MDパス、チャンク一覧を取得する
log_agent_run AI社員の稼働ログを記録する
submit_classification_label 作業中に見つけた分類ラベルを候補として記録する
submit_knowledge_promotion_candidate ナレッジ化・スキル化・社員化候補を送る
submit_eval_case 評価テスト化すべきケースを登録する

この時点では、MCPから knowledge/ を直接書き換えない。既存ルール通り、ナレッジ管理担当と田中祐一の確認を挟む。

新プロジェクトにするかどうか

現時点の推奨は、新しいSupabaseプロジェクトを作らず、TanakaKnowledgeOSに統合すること。

理由:

  1. 既存の認証、アクセス権限、MCP、ナレッジ更新ループを使える。
  2. mcp_tool_events や knowledge_manager_handoffs と自然に接続できる。
  3. 田中祐一AI、添削、制作、QA、個人コンテキストが同じOS内でつながる。
  4. 別プロジェクト化すると、データ同期と権限管理の二重化が起きる。

新プロジェクトを検討する条件:

  1. 外部企業向けに独立SaaSとして販売する。
  2. 田中祐一AIとは別の認証、課金、テナント管理が必要になる。
  3. 受講生データと社内AI社員育成データを物理的に分離したい。
  4. 現在のTanakaKnowledgeOSのRLSやテーブル設計では安全に共存できない。

今後の運用ルール

今後は、作業終了時に毎回すべてを手作業で「スキル化・ナレッジ化・社員化」しない。代わりに、DBへ候補を投げる。

作業後に残すもの:

  1. これは判断基準か。
  2. これは事例か。
  3. これは失敗パターンか。
  4. これはAI評価テストに使えるか。
  5. これは特定AI社員のスキル、ナレッジ、ルールに昇格すべきか。

この5つを classification_labels と knowledge_promotion_queue に残せば、週次のナレッジマネージャーが判断できる。毎回その場で完璧に整理しきる必要はない。

重要な設計思想

DB化の目的は、Markdown整理を否定することではない。

目的は、Markdownだけでは追えなかった以下を追えるようにすること。

  1. どの原本から来た知識か。
  2. どの評価軸で分類されたのか。
  3. 誰、またはどのAIが分類したのか。
  4. どのAI社員が使ったのか。
  5. 田中祐一が承認したのか。
  6. 後から評価軸が変わったとき、何を再分類すればよいのか。

過去の仕分け作業は、DB化によって「消える」のではなく「比較できる履歴」になる。ここを先に設計しておけば、今後は評価軸が変わるたびにゼロからやり直す必要がなくなる。

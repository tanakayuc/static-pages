B4: 哲学（哲学・原理）を本番AIへ常時ロードする設計

作成日: 2026-06-14
担当: Claude設計＋実装 / Codex検証 / 田中デプロイ承認
ステータス: 実装中

0. これは何の問題か（1行）

田中祐一AIの添削に 哲学・原理（OS核）が届いていない。MCP経路で捨てられているため、「マーケの正解」ではなく「田中祐一だったらどう考えるか」を再現する土台が欠落していた。

1. 根本原因（実査で確定）

添削ナレッジは knowledge/_bundle_添削モード.md（約581k字）に集約 → MCPの buildCompactKnowledge()（apps/mcp/src/mcp/tools/correction.ts）で18,000字に圧縮して返す。

圧縮ロジックが:
1. バンドルを見出し単位でセクション分割
2. キーワードでスコアリング → score=0のセクションを捨てる
3. 上位10セクションだけ残す

→ 哲学ファイル（例: 疑いのレベル_市場の成熟度・マーケティングは教育である）は添削用キーワード（LP/セールス/CTA等）に当たらず score=0 → 捨てられる。バンドル先頭に置いても無意味だった。

経路は2つ・問題は片方だけ

経路 哲学の状態
MCPツール（Cursor/Claude/Codex/ブラウザ＝本命の4surface） buildCompactKnowledgeで欠落 ← 修正対象
RESTルート（Web UI /c/、サーバー側LLM呼び出し） full bundleをそのまま使う＝既に届いている

2. 設計（3階層ロード）

buildCompactKnowledge を3階層に再構成する。

  • Tier0（哲学・原理）= 常時ロード。スコアに関係なく必ず含める。バンドル先頭の専用ブロックから抽出し、出力の先頭に「## 田中祐一OS 核（哲学・原理）｜常時適用」として配置。
  • Tier1（pinned）: 既存のモード別pinnedパターン（17ステップ等）。
  • Tier2（keyword）: 既存のキーワードスコア上位。

バンドル側（02_ツール/開発支援/統合添削_bundle.py）

tier0ファイル群（既に先頭に配置済み）を機械可読センチネルで囲む:

<!-- ★TIER0-START★ OS核（哲学・原理）｜常時ロード領域 -->
…哲学・原理ファイル本文…
<!-- ★TIER0-END★ -->

※コメントなので見出し分割に影響せず、tier0が無ければセンチネルも出ない（後方互換）。

MCP側（buildCompactKnowledge）

  1. センチネル間を抽出 = tier0ブロック。残りを rest とする。
  2. tier0が TIER0_BUDGET(既定11,000字)以内ならブロック全文を常時採用。超えたらtier0内でキーワード順に予算まで採用（哲学が増えてもスケール）。
  3. rest に対して既存のpinned＋keyword選抜（重複防止のためtier0は除外済み）。
  4. 全体上限 KNOWLEDGE_RETURN_LIMIT を 18,000→24,000字 に引き上げ（哲学10k＋ルール4k＋ジャンル別10kが共存できる量。2026-06-14 田中承認）。
  5. 両定数とも env で上書き可（KNOWLEDGE_RETURN_LIMIT / KNOWLEDGE_TIER0_BUDGET）。

後方互換・反映タイミング

  • バンドルはGitHubから実行時取得（cache TTL 300秒）→ push後5分で反映、再デプロイ不要。
  • buildCompactKnowledge のコード変更は apps/mcp の再デプロイが必要。
  • センチネル有無で安全に縮退（旧バンドル×新コード／新バンドル×旧コードのどちらでも壊れない）。

3. トークンコスト（透明性）

24,000字 ≈ 約10〜13kトークン/添削1回。受講生のClaude/Cursor契約で消費。精度優先のNorth Starに基づき田中承認済み。重い場合は env で18,000等に即ダイヤルダウン可。

4. 検証

  • 既存テスト（tests/correction-knowledge.test.ts）が壊れないこと（センチネル無しbundle＝従来動作）。
  • 追加テスト: キーワード0当たりの哲学ブロックが出力に必ず残ること（本修正の核）。
  • apps/mcp type-check / vitest。
  • Codex独立レビュー。

5. デプロイ手順（田中／技術担当）

  1. （Claude）bundle.py改修・バンドル再生成・correction.ts改修・テスト → commit/push（哲学が5分後にバンドル反映）
  2. （田中/技術）apps/mcp を本番デプロイ（コード変更反映）
  3. （検証）添削モードを起動し、出力先頭に「核（哲学・原理）」が乗ること・疑いのレベル/マーケ=教育が参照されることを確認

6. Codexレビュー反映（2026-06-14・コミット544678504をレビュー）

致命的=なし。要修正3点・任意2点を反映済み（追いコミット）:

  • センチネル抽出の堅牢化: indexOf先頭一致→行アンカー正規表現で「ちょうど1組」検証。欠落/複数/逆順/本文偶然一致は安全に tier0='' へ縮退。
  • 超過時の挙動変更: tier0が予算超過した時、キーワード選抜だと「当たらない哲学が落ちる」ため、文書順（キュレーション順＝優先度）で予算まで詰める方式に変更。哲学の取りこぼし事故を構造的に排除。
  • env下限ガード: KNOWLEDGE_RETURN_LIMIT は下限18,000・負数/NaN無効化（過小設定で哲学が末尾カットされるのを防止）。TIER0_BUDGET も下限6,000。
  • 数値の訂正: 実バンドルtier0は wc -cで10,976バイトだが、JS .length（文字数）では約4,955字。予算11,000に対し2.2倍の余裕があり当面安全。
  • 任意（新バンドル×旧コードでセンチネル漏れ）: デプロイ手順で「コード反映を急ぐ」ことで実害最小化。HTMLコメントなので旧コードでも基本は見出し前に落ちる。

テスト: robust系2件追加（複数センチネル縮退/偽センチネル耐性）。計6 passed + 実バンドル全モードでセンチネル漏れ無し・哲学常時ロード確認。

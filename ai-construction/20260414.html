# AI社員化プロジェクト — 作業背景と現状

## プロジェクトの目的

定常業務をAI社員（エージェント）に移植し、チームで育てていく仕組みを構築する。

「AI社員を育てる = ナレッジを育てる」という考え方で進めています。


## 新設したフォルダ構造

2026年4月14日に以下の4フォルダを新設しました。

```
theleadpromotion/
├── knowledge/          ← 再利用可能なナレッジを集約（受講生への配布OK）
├── agents/             ← AI社員の定義ファイル（受講生への配布OK）
├── configs/            ← 案件ごとのマスターファイル（配布禁止）
└── skills/             ← スキル定義（.cursor/skills/ から移植・Claude Codeでも使える）
```

既存フォルダはデータ保存庫としてそのまま残しています。ファイルの移動は最小限。


## 各フォルダの役割

### knowledge/

「何が正しいか」を集めた唯一の真実の場所。
案件データは入れない。普遍的に再利用できる原則・手順・判断基準だけを入れる。

```
knowledge/
  共通/               コピーライティング原則・わしづかみコンセプト・用語辞書・悪魔の代弁者
  ファネル型/          4本動画構成・1本動画構成・5チャレンジ構成・自動化ウェビナー構成
  スキル別/            LP作成・LP添削・セールス分析・議事録作成・ステップメール作成など
  媒体別/              YouTube・Meta広告・TikTok・ブログ・SNS
```

現時点のファイルはすべてプレースホルダーです。
「ここに書く」という場所を作った状態。実際のノウハウはStep4で移植予定。

### agents/

AI社員の定義ファイル。`agent.md` に役割・判断基準・参照するナレッジを書く。

```
agents/
  議事録担当/         MTG書き起こし → 議事録 → Chatwork通知
  セールス分析担当/   セールス書き起こし → 分析・フィードバック生成
  ナレッジ管理担当/   データスキャン → ナレッジ抽出 → 整合性チェック（全体の指揮者）
  LP添削担当/         LP原稿を受け取り → 添削フィードバック生成
  LP作成担当/         configとナレッジを読んで → LP原稿を0から新規作成
```

LP作成とLP添削は別の社員として分けています（インプット・アウトプットが違うため）。

### configs/

案件ごとのマスターファイル。「田中さんのマスターファイル」のイメージ。

スキルやエージェントはこのファイルを読んで「誰の案件か・どのファネル型か・
どのナレッジを使うか・データはどこにあるか」を判断します。

```
configs/
  プロモ案件/
    たかみず.md         人格プロフィール・ファネル構成・データの場所・使用ナレッジ を記載済み
    （他案件は今後追加）
  田中祐一/
    マスター.md         自社プロジェクト全体（A〜Eファネル）を定義済み
    （PLCプロモ・非属人チャンネルなどは今後追加）
  受講生/              （今後追加）
  自社事業/            （今後追加）
  チャンネル/           （今後追加）
```

### skills/

```
skills/
  loom分析/            ← .cursor/skills/loom-analyze から移植
  loom同期/            ← .cursor/skills/loom-sync から移植

  LP作成/              ← 新規作成
  LP添削/              ← 新規作成
```

`.cursor/skills/` はCursorが直接参照するため残しています。
`skills/` はClaude Codeからも使えるようリポジトリ直下に置いた場所です。


## 設計上の重要な決定事項

### データ保存庫はそのまま残す

`21_プロジェクト一覧/` `11_受講生サポート/` などの既存フォルダは「データ保存庫」として存続。
ファイルを移動しない。configで「データの場所はここ」と指さすだけ。

### ナレッジは1箇所にだけ書く（Single Source of Truth）

ナレッジを複数箇所にコピーすると片方が古くなる。
`knowledge/` に書いて、社員仕様書（agent.md）やスキル（SKILL.md）はそこを参照するだけ。

### 確認ゲートを最初は全部つける（Supervised Autonomy）

エージェントの実行フェーズ：
- 見習い: 毎ステップ確認を求める
- 半人前: 完了後に報告・差し戻し可
- 一人前: 自動実行・異常時のみ通知
- 熟練: 完全自動・ログのみ

同じタスクを「修正なし」で10回連続通過したら確認ゲートを外す。

### configは「案件」単位ではなく「プロジェクト」単位

採用・講座運営・AI構築なども全部プロジェクトとして `configs/自社事業/` で管理。
人物ではなくプロジェクト（チャンネル名など）でconfigを作る。

### LP作成とLP添削は別スキル・別エージェント

| | LP作成 | LP添削 |
|---|---|---|
| 対象 | 白紙から作る | 既存LPをレビューする |
| インプット | config + ナレッジ + 素材 | 完成または下書きのLP |
| アウトプット | LP原稿（新規） | フィードバックコメント + 修正案 |

## スタッフへの指示方針

「AIを勉強してください」ではなく「AI社員を1人育ててください」という課題を与える。

スタッフへの具体的な指示：
```
あなたが担当している業務の中で「毎週必ず発生する作業」を1つ選んでください。
その作業を担当するAI社員を1人作ってください。
以下の3つを書けば完成です：
1. この社員は何をする人か（1〜2文）
2. 作業の手順（箇条書き）
3. うまくいったかどうかの判断基準
```

田中さんが先に「議事録担当」「セールス分析担当」の2体を手本として作り、
「これを参考に各自1体作ってね」という形にするのが最も効果的。

チームの役割分担：
- (a) 社員育成担当: 業務知識をagent.mdに書く人（AIに詳しくなくていい）
- (b) 基盤構築担当: エージェントが使うツール・MCP連携を作る人（技術担当）

設計：会員サイト背骨の高精度格納＋オーバーフロー回避取得＋DB移植を見据えて（2026-06-16）

田中の指示（2026-06-16）：会員サイト（メイン講義＝プロダクトローンチ体系の背骨）は最重要かつ大容量。受講生はこの根幹の上でグルコン/添削を受けている。土台がAI側に無いのはおかしい。これを「引用しやすく・容量をオーバーフローさせず・綺麗に抽出できる仕組み」で格納し直す。DB移植も見据えた設計にする。

問題の核心

背骨は大容量（メイン講義77レッスン＋他11プログラム）。現アーキテクチャ（フラットbundle全fetch → buildCompactKnowledgeがkeyword/n-gramで約24k字選抜）では：

  1. オーバーフロー：背骨を全部knowledge/に入れるとbundleが肥大。tier0予算（11,000字）はすぐ超える（先日13,344字で視点のマッピングが落ちた実例）。
  2. そもそも浮上しない：keyword/n-gram選抜では、字面一致が強くても大プールの競合で関連背骨が選ばれない（golden baseline B群＝組織体制/経営が浮上せず、見出しブーストも回帰で撤回＝通算4回目の実証）。

→ 「全部載せる」ことも「キーワードで選ぶ」ことも、背骨の規模では破綻する。

設計原則（オーバーフロー回避の本質）

AIに渡す量は総量に依らず一定にする。「意味で近い背骨だけ top-K」を渡す。

  1. 正本＝git上の構造化レコード（章アンカー付き・引用しやすい・DB移植可）。
  2. 検索＝embedding（意味検索）でtop-K選抜。総コーパスがどれだけ増えても、クエリに意味的に近い少数だけ予算内で注入＝オーバーフローしない。これがキーワード検索の限界（B群）を根治する。
  3. tier0は最小（普遍的な哲学/原理のみ・常時）。背骨は「常時ロード」でなく「関連時にembeddingで取得」。
  4. DB移植は派生：構造化レコード（正本・git）→ build → ①実行用index ②embeddingベクトル。ベクトルはローカル or Supabase pgvector。正本は最後までgit（差分レビュー/Codex/AI編集の資産維持）。

背骨レコードのスキーマ（引用しやすさ＝アンカー＋出典）

既存frontmatter＋matrix＋背骨アンカーを持たせる：

---
type: 哲学|原理|判断基準|基準値|事例|語彙
program: PLC会員サイト        # どのプログラムか
chapter: 8                     # 章（体系の背骨位置）
lesson: 8-3                    # レッスンID（安定・引用の宛先）
backbone: true                 # 背骨フラグ（体系の軸）
主張: <一文>
立場/活用シーン/事例link/リフレーム: ...（matrix）
出典: "メイン講義 8-3 書き起こし原本#L<行>"   # 原本まで遡れる
status: 採用
---
  • 引用しやすさ：program/chapter/lesson で「8章ローンチ運営の入金率基準」と体系位置で引ける。出典で原本まで遡れる。
  • 体系（面）の保持：chapter順がそのまま脳の背骨。点（グルコン/添削）は将来 backbone_ref: 8-3 で背骨に紐付ける。

取得の仕組み（オーバーフローしない）

クエリ
 → ① tier0(普遍哲学・常時・最小)
 → ② embedding近傍検索で関連レコードtop-K（背骨含む・予算内）
 → ③ keyword/n-gramは補助シグナル（完全一致の僅差調整のみ）
 → AIに渡す量は常に一定（予算24k等）。総量が増えても変わらない＝オーバーフロー回避

DB移植を見据えた段階設計

Phase 内容 DB移植との関係
B0 スキーマ確定 背骨アンカー付きレコード形式を確定。lint拡張（chapter/lesson/backbone検証） レコード＝将来のDB行。今のうちに構造を整える
B1 embeddingパイプライン レコード→チャンク→ベクトル生成（モデル選定）。ローカルindex（JSON/sqlite-vss等）でまず動かす ベクトルは正本から再生成可能な派生
B2 retrieval接続 buildCompactKnowledgeをembedding top-K選抜に接続。golden baseline A群で無回帰検証必須（わしづかみ等が浮上し続ける）。B群が✅化することがゲート —
B3 背骨ingest 会員サイト背骨を段階的にレコード化（章順）。embeddingで取得されることを都度確認 レコード増＝DB行増。retrieval量は一定
B4 (条件付き) pgvector 同時多重・大規模・横断クエリが実害化したらSupabase pgvectorへベクトル/レコードを投影 正本はgitのまま。DBは派生・読み取り

重要な判断（田中の確認が要る大きな分岐）

  • embedding着手＝Phase B本体。これは「本番AIの取得を変える」構造判断。コスト（embedding API・ベクトル保存）が発生。
  • embeddingモデル：OpenAI text-embedding-3-small等（多言語・安価）か、ローカルモデルか。日本語性能とコストで選ぶ。
  • まずローカルindexで動かし（B1-B3）、規模が要求したらpgvector（B4）＝段階的でコスト最小。

まとめ

背骨の大容量が、ついにembeddingを「あったらいい」から「無いと土台が載らない」に変えた。設計の肝＝正本はgit構造化レコード（章アンカー・引用容易・DB可）／取得はembedding top-Kで総量に依らず一定（オーバーフロー回避）／tier0最小／DBは最後に派生。これで会員サイト背骨を高精度かつ破綻なく載せられる。

設計: Zoom一括蒸留 Stage B仕分けルール

作成日: 2026-07-03

現在地

2026-07-03時点で、抽出済みZoom YAML 320本は knowledge/ のStage B全件統合MDへ反映済み。

  • 個人コンサルYAML: 281本
  • グルコンYAML: 39本
  • 生成エントリ: 10,997件
  • 出力: 哲学 / 原理 / 判断基準 / 基準値 / 事例 / 用語と話法 / 実験宣言と採用見解
  • bundle/index/embedding: 再生成済み

この全件統合MDは「AIが参照できるようにするためのライブ参照面」であり、最終的な代表ルール集ではない。既存の手動キュレーション済みknowledgeを優先し、全件統合MDは追加根拠・未整理発話の探索に使う。

棚分け

1. ライブ参照

対象: Stage A YAMLから自動生成した全件統合MD。

条件:
- quote と line がある。
- Stage A抽出で田中発話として拾われている。
- 既存bundleへ入る type と status: 本文 を持つ。

扱い:
- AIが検索できる。
- ただし、回答で断定する前に既存の手動キュレーション済みknowledgeと照合する。
- 重複・表現揺れ・案件固有性は残っていてよい。

2. 採用候補

対象: ライブ参照から、人手で代表ルールへ昇格できそうなもの。

条件:
- 判断テスト: 田中の良し悪し判断・条件分岐・基準値として使える。
- 移転テスト: 当該案件以外にも使える。
- 出典テスト: YAMLのquote/lineから原本Zoomへ戻れる。
- 帰属テスト: 田中本人の発話、または endorsed_views として田中の採用が分かる。

扱い:
- 既存knowledgeの該当ファイルへ統合する前に、類似ルールの重複を探す。
- 数値は文脈を添えて記録し、絶対基準にしない。

3. 手動統合採用

対象: 既存の哲学・原理・判断基準・基準値・事例ファイルへ統合したもの。

条件:
- 代表表現に統合済み。
- 根拠YAMLとlineを明示済み。
- 既存ルールと矛盾しない、または矛盾の扱いを注記済み。

扱い:
- AI回答で優先する。
- 全件統合MDより上位の正本として扱う。

4. 仮説・保留

対象:
- 話者帰属が弱いもの。
- 田中の考えが後日変わっている可能性が高いもの。
- own_experiments のうち結果確認前のもの。
- endorsed_views のうち明示同意が弱いもの。

扱い:
- knowledge/_inbox/ へ置く。
- status: 仮説 にしてbundle除外する。
- 追加根拠が出たら採用候補へ戻す。

5. 棄却・アーカイブ

対象:
- 単なる相づち、事実確認、案件固有すぎる作業指示。
- コンプライアンス上、AIに持たせるべきでない詳細。
- 音声認識崩れで意味が確定できないもの。

扱い:
- 本番knowledgeへ入れない。
- 必要なら理由だけを台帳に残す。

粗仕分けカテゴリ

全件統合生成では、以下のキーワードカテゴリで粗く並べる。

  • 広告・数値検証
  • ファネル・ローンチ
  • セールス・オファー
  • コンセプト・コピー
  • 講座・運営・コミュニティ
  • 事業設計・ライフスタイル
  • AI・仕組み化
  • その他

この分類は探索用であり、最終のknowledge配置を決めるものではない。最終配置は、AIがどう使うかを基準に knowledge/共通、knowledge/スキル別、knowledge/ファネル型、knowledge/事例 へ振り分ける。

優先順位

  1. 既にAI回答で使われやすい領域を先に整える。
  2. 広告・数値検証
  3. ファネル・ローンチ
  4. セールス・オファー
  5. 既存の手動統合5ファイルに近いものから代表ルール化する。
  6. own_experiments / endorsed_views は急いで正本化しない。
  7. 1回で完璧に仕分けようとせず、日次同期でAI反映を維持しながら少しずつ代表ルールへ圧縮する。

日次反映ルール

  1. Stage Aで新しいYAMLが増える。
  2. 日次同期がYAML digestの変化を検知する。
  3. Zoom蒸留_StageB全件統合生成.py --write が全件統合MDを再生成する。
  4. ナレッジ再生成.sh が bundle/index/embedding を再生成する。
  5. 田中AIナレッジ同期_最新状況.md で未統合0件とmd5一致を確認する。

既知の注意点

  • 7本のStage A YAMLは、未クォートのASCIIコロンが残っており、そのままではPyYAMLが落ちる。Stage B生成スクリプトは読み込み時だけ自動補正している。
  • 全件統合MDは大きい。現時点のbundleは約9.2MB。検索精度や運用負荷を見ながら、手動統合済みの代表ルールへ段階的に圧縮する。
  • Stage A全体は627本対象で、現時点では320本がYAML化済み。残りが抽出されるたび、日次同期でStage B反映される。

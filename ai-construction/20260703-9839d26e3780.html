Codex引き継ぎ指示書：Zoom一括蒸留（Stage A）の継続

作成日: 2026-07-03（Claude Sonnet 5 → Codex への引き継ぎ）
田中の意向: 時間はかかってよいので、ゆっくり・低コストで完走させたい。品質（仕分けの考え方）は今までの水準を落とさない。

0. これは何の作業か（背景）

田中祐一AI（knowledge/ 配下・_bundle_添削モード.md経由でMCPが配信）の「思考層」を厚くするため、Zoomの個人コンサル506本＋グルコン121本＝計627本の書き起こしから、田中祐一固有の哲学・原理・判断基準・基準値を抽出している。

⚠️最重要の誤解注意（田中に一度説明済み）: この作業（Stage A）が終わっただけでは田中祐一AIは1ミリも賢くならない。

  • Stage A（今回引き継ぐ作業）: 生の書き起こし(md) → 思考抽出YAML（13_データ原本保存庫/zoom*/_抽出/*.yaml）。これは中間生成物で、AIは読まない。
  • Stage B（別途・このドキュメントの対象外）: YAML → knowledge/*.md（frontmatter付き）→ 統合添削_bundle.pyで再生成 → 初めてAIが読める。
  • Stage Bは別セッションで実施予定。CodexはStage Aのみを担当する。

1. 現在の進捗（2026-07-03時点）

  • 総対象: 627セッション（個コン506・グルコン121）
  • 完了（YAML作成済み・disk上に実在）: 320本（git ls-files で確認可能。git管理下）
  • 確定スキップ（録画0分・純事務連絡）: 4件。一覧は 97_ザ・リード事業/AI構築/一括蒸留_確定スキップ一覧_20260703.json
  • 残り: 約368件（個コン286・グルコン82）。厳密な最新値は下記コマンドで確認すること（進行中に他エージェントが処理する可能性があるため、着手直前に必ず再取得）:
    bash python3 02_ツール/Zoom抽出/次バッチ割当生成.py --status

2. コスト方針（最重要・田中の明示指示）

  • モデルはOpus級を使わない。Claude Code / Codex利用時は中間グレード（Sonnet相当）を選ぶこと。極端に安いモデル（Haiku級）も使わない＝「品質を保ちつつ燃費を良くする」帯を狙う。
  • 前回、Opus級で$200分を使って320本しか進まなかった（1本あたり約$0.6〜2.8とばらつくが、平均して高すぎた）。原因は「5時間級の生書き起こしを丸ごと読んで全引用を検証する」重い処理をOpusでやっていたこと。
  • 時間がかかるのは許容。急がず、コストを抑えて完走することを優先する。
  • 並列数は多くしすぎない（2〜4体目安）。Claude Code側は5体並列でAPIエラー（一時的な接続断）や月次上限超過を経験している。

3. 作業手順（1バッチあたり）

3.1 次にやるセッションを取得する

cd /Users/tanakayuichi/Projects/theleadpromotion
python3 02_ツール/Zoom抽出/次バッチ割当生成.py --corpus personal --size 8 --out /tmp/割当_P.json
# または
python3 02_ツール/Zoom抽出/次バッチ割当生成.py --corpus group --size 5 --out /tmp/割当_G.json
  • 個コンはバッチサイズ8〜12件、グルコンは長尺（60〜300分）なので5件が目安（過去の実績: グルコン5件で1エージェント2.5〜3時間程度）。
  • 出力JSONの各要素: session_id / corpus / files（原本パス配列。リポジトリルート相対）/ date / topic。

3.2 スキーマを読む（必読・正本）

  • 02_ツール/Zoom抽出/抽出スキーマv2.md
  • v1〜v2: session/opening/diagnosis_flow/recommendation/speaking_style/corrections_made
  • v2（5要素モデル対応・最重要）: philosophy_utterances / principle_utterances / criteria_utterances / benchmark_utterances / vocabulary / case_record
  • v2.1: referenced_cases（田中の持ちネタ事例）/ worldview_utterances（時代認識）/ script_demos（口述デモ）/ closing
  • v2.2追補（2026-07-02・必ず読むこと）: own_experiments（田中の「これから試す」宣言）/ endorsed_views（他者の見解に田中が同意・採用したもの）
  • お手本YAML（構造を丸ごと真似る）:
  • 個コン: 13_データ原本保存庫/zoom個人コンサルデータ/_抽出/personal_20260515_a3a3d49c4e47.yaml
  • グルコン: 13_データ原本保存庫/zoomグルコンデータ/_抽出/group_20250619_008c158c6acd.yaml（相談単位の分割例として優良）

3.3 抽出する（1セッションずつ）

各セッションについて：
1. files の原本md（複数ある場合は全部）を読む。
2. スキーマ全セクションをverbatim（一字一句改変禁止）で抽出。
3. 出力先: 個コン→13_データ原本保存庫/zoom個人コンサルデータ/_抽出/<session_id>.yaml、グルコン→13_データ原本保存庫/zoomグルコンデータ/_抽出/<session_id>.yaml。
4. 出力前に同名YAMLが既に存在する場合は上書きせずスキップする（他エージェントとの競合防止）。
5. 1セッション書き終えるごとに保存（大きな作業を1つのトランザクションにしない。中断されても既存分は残る）。

3.4 対象外（田中発話比率が低いセッション）の扱い（田中裁定 2026-07-02）

  • 索引が「対象外」と分類したセッション（田中発話比率<40%の社内実務MTG等）も捨てずに「軽め抽出」する。
  • 軽め抽出＝田中の発言だけを拾い読みし、市場観・組織判断・実験宣言など光るものだけ抜く。薄いYAMLで良い。
  • YAML冒頭に # 軽め抽出（対象外分類・田中裁定2026-07-02による） とコメントを入れる。
  • 完全スキップして良いのは「15分未満の純事務連絡・録画0分・純雑談」のみ。それ以外は軽めでも拾う。

3.5 品質ルール（厳守）

  1. verbatim＋行番号必須。quote は原本のその行の連続部分文字列であること（複数行にまたがる引用は禁止。文脈が長い場合は context 側に逃がす）。
  2. YAML書式注意: 引用文にASCIIコロン(:)が含まれる場合は必ずクォートする（quote: "〜:〜"）。過去、これが原因でYAMLパースエラーが頻発した。
  3. 固有名詞は 13_データ原本保存庫/_固有名詞辞書.json と照合し、補正した場合は各YAML末尾に correction_log として記録する（引用原文は改変しない。ログに「誤認識→正表記」の形で残す）。特定不能な人名は「要確認」と明記（勝手に確定しない）。
  4. 判断テスト・移転テストを適用: 単なる相づち・事実確認は拾わない。案件固有の事情は recommendation/case_record の文脈としてのみ記録し、philosophy/principle には昇格させない。
  5. 田中以外の発話を田中の哲学として混入させない。話者ラベルが崩壊している原本（会場マイク収録・複数話者が「田中 祐一」ラベルに誤割当されるケースが頻発する）では、一人称・内容の文脈で田中発話を同定し、判別不能なものは除外して extraction_meta にその旨を記す。
  6. 他者の見解に田中が同意・採用した場合は philosophy_utterances に混ぜず endorsed_views に分離する（speaker/quote/endorsement_quote/line）。
  7. コンプライアンス上センシティブな内容（法規制のグレーゾーン、他社批判、個人の契約トラブルの詳細等）は抽出対象外とし、extraction_meta にその旨だけ記録する（内容は書かない）。

3.6 検証（提出前に必ず実行）

python3 02_ツール/Zoom抽出/verify_quotes.py 13_データ原本保存庫/zoom個人コンサルデータ/_抽出/<作成したファイル>.yaml
  • 全quoteの「行番号突合」を自動チェックする。NGが出た項目は quote または line を修正し、NG 0になるまで直す。
  • 併せて python3 -c "import yaml; yaml.safe_load(open('<path>'))" でパース確認（verify_quotes.py内でも自動でパースするため、正常終了すれば重複チェック不要）。

3.7 保全（コミット・push）

作業単位ごと（1バッチ完了ごと）に必ずコミットしてpushする。他のエージェント・セッションが同時にmainへpushしていることがあるため、push前に必ずpullし、rejectされたら再試行する。

cd /Users/tanakayuichi/Projects/theleadpromotion
git add 13_データ原本保存庫/zoom個人コンサルデータ/_抽出/ 13_データ原本保存庫/zoomグルコンデータ/_抽出/
git commit -m "C(一括蒸留): <バッチ内容の要約>本を保全

Co-Authored-By: Codex <noreply@openai.com>"
git pull --rebase origin main
git push origin main
  • pushが non-fast-forward で拒否されたら、git pull --rebase origin main → git push origin main を再試行する。
  • 知識ベース（knowledge/）や configs/ には一切書き込まない（Stage Aの範囲外）。
  • bundle/index/embeddingの再生成もしない（Stage B担当の別セッションが行う）。

3.8 次バッチへ

3.1〜3.7を繰り返す。次バッチ割当生成.py --status の残数が0になったら個コン・グルコンとも完了。

Stage A（このドキュメントの作業）が全て完了したら、97_ザ・リード事業/AI構築/一括蒸留_進捗台帳_20260702.md に完了報告を追記し、田中にStage B（YAML→knowledge統合→bundle再生成→ライブ検証）着手の可否を確認する。Stage Bは別途、以下を参照して進める設計になっている:
- 97_ザ・リード事業/AI構築/設計_構造化レコード＋embedding＋Q&Aループ_20260616.md
- ゲート設計: verbatim根拠＋既存重複チェックを通過したもののみ status: 採用（ライブ配信）。確信不足・ゲスト帰属曖昧・考えが変わっていそうなものは status: 仮説（knowledge/_inbox/ へ・バンドル除外の隔離棚）。

5. 参考：これまでの実績データ（見積もりの参考）

  • Sonnet/Opus問わず、個コン1本あたりの通常回は40〜60分の書き起こしで概ね処理可能。グルコンは複数相談を含み60〜300分と重く、1バッチ5本が目安（重い回は1本で個コン1.5〜2倍の工数）。
  • 品質は毎バッチ verify_quotes.py でNG 0まで追い込むこと。過去、初回検証でNGが数件〜十数件出るのは通常なので、修正して再検証するプロセスを1回で終わらせず徹底する。
  • 固有名詞辞書（13_データ原本保存庫/_固有名詞辞書.json）は972エントリ（2026-07時点）。無ければ都度確認し、勝手に新規登録はしない（要確認のままYAMLに残す）。

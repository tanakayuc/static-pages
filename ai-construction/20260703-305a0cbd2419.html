田中祐一AIとAI社員の連携設計メモ

作成日: 2026-07-03

1. 先に結論

田中祐一AIとAI社員は、絡ませた方がよい。

ただし、ユーザーに「どのエージェントを呼びますか」と毎回選ばせるのではなく、田中祐一AIを総合窓口兼PMとして置き、内部で必要なAI社員を呼び分ける構造がよい。

田中祐一AIは「田中祐一の脳を再現する本体」。
AI社員は「その脳の中にある専門部署・専門担当」。
スキルは「担当者が行う作業手順」。
ナレッジは「担当者が参照する判断材料」。
outputは「担当者が残す成果物と検証ログ」。

この関係にすると、田中祐一AIの出力は次の3点で強くなる。

  1. 精度: 必要なナレッジだけを専門担当が読むため、判断が薄まりにくい
  2. 独自性: 田中祐一メソッド、発話、添削、事例を担当別に蒸留できる
  3. 再現性: 毎回同じ工程、同じ確認ゲート、同じ成果物形式で動かせる

2. 現在地の整理

既存の agents/ には、すでに以下のような担当がある。

  • 議事録担当
  • セールスフィードバック担当
  • ナレッジ管理担当
  • ファネル分析担当
  • LP添削担当
  • LP作成担当
  • コンテンツライティング担当
  • 経理担当
  • 採用担当

これはかなり良い初期設計。

一方で、現在の課題は、エージェントが「部署」ではなく「個別担当の平置き」に近いこと。

そのため、マーケティングのような広い仕事では、結局田中さん本人がPM、設計者、コンセプター、ライター、レビュー担当を兼ねてしまう。

教材の考え方に寄せるなら、次のように変える。

田中祐一AI
  └── AI部署
        └── AI社員
              └── skill
                    └── knowledge / output

3. 田中祐一AIの役割

田中祐一AIは、単体の万能回答AIではなく、以下の役割を持つ。

3-1. 総合窓口

ユーザーの依頼を受ける。

例:

  • コンセプトを作りたい
  • LPを添削してほしい
  • ステップメールを書きたい
  • ファネル全体を見直したい
  • 受講生向けに説明してほしい

3-2. ルーター

依頼内容を見て、どの部署、どの社員、どのスキルを使うか決める。

例:

「LPを作りたい」
  → Marketing Department
  → lp-architect
  → 01-lp-structure-design
  → 必要なら proof-researcher / copywriter / qa-reviewer へ回す

3-3. PM

複数のAI社員を順番に動かす。

同時にたくさん呼ぶのではなく、工程に沿って回す。

ビジネス設計
  → ファネル設計
  → コンセプト
  → オファー
  → シナリオ
  → LP
  → ステップメール
  → セールス
  → QA

3-4. 最終編集者

AI社員が出した成果物を、田中祐一AIとして統合する。

担当者ごとの出力をそのままユーザーに出すのではなく、最後に田中祐一AIが、文体、判断、温度感、説明順を整える。

4. AI社員を細分化する基準

細分化はした方がよい。

ただし、増やせば増やすほど良いわけではない。

新しいAI社員として独立させる条件は、次の5つ。

  1. その作業が何度も発生する
  2. 入力と出力が明確に分けられる
  3. 参照するナレッジが他の担当と違う
  4. 品質基準やレビュー観点が違う
  5. スタッフや受講生にも将来渡せる

この5つを満たすなら社員化する。
満たさないなら、既存社員の skills/ に入れる。

例:

  • LP作成担当をいきなり10人に分ける必要はない
  • まずは LP作成担当の中に複数skillを持たせる
  • 何度も独立して使うようになったら、LP構造設計担当、LPコピー担当、LPデザイン担当に分ける

5. マーケティング部署の推奨構造

最初に作るべき部署は、Marketing Department。

田中さんが自分で巻き取ってしまいやすい領域なので、ここを社員化すると効果が大きい。

Marketing Department/
├── _RULE.md
├── cmo/
│   ├── _RULE.md
│   └── skills/
│       └── 01-marketing-task-router/
├── business-design-strategist/
├── funnel-architect/
├── concept-maker/
├── offer-architect/
├── scenario-planner/
├── lp-architect/
├── stepmail-writer/
├── proof-objection-researcher/
├── marketing-copywriter/
└── marketing-qa-reviewer/

cmo

マーケティング依頼の受付、担当振り分け、工程管理を行う。

田中祐一AI本体から最初に呼ばれる部署内PM。

business-design-strategist

単発ファネルではなく、事業全体、商品群、プロモーション順序、年間導線を見る。

funnel-architect

1つのプロモーション内のファネルを設計する。

チャレンジ、動画、ウェビナー、個別相談、説明会、直売りなどの組み合わせを判断する。

concept-maker

3C、上位概念、共通の敵、切り口、入口コンセプトを作る。

offer-architect

商品設計、特典、価格、保証、限定性、比較軸を作る。

scenario-planner

動画、セミナー、価値提供、教育コンテンツの流れを作る。

lp-architect

LPの構造、見出し、セクション順、CTA、証拠配置を設計する。

stepmail-writer

ステップメール、LINE配信、価値提供メッセージを作る。

proof-objection-researcher

お客様の声、実績、反論、不安、FAQ、信頼材料を整理する。

marketing-copywriter

見出し、本文、メルマガ、広告文、SNS投稿などのコピーを作る。

marketing-qa-reviewer

田中祐一メソッドとして成立しているか、ファネルの矛盾がないか、訴求がズレていないかを確認する。

6. 既存エージェントとの対応

既存のエージェントは捨てない。

Marketing Departmentの中に位置づけを与える。

既存: ファネル分析担当
  → Marketing Department/funnel-architect または funnel-analyst

既存: LP作成担当
  → Marketing Department/lp-architect

既存: LP添削担当
  → Marketing Department/marketing-qa-reviewer または lp-reviewer

既存: コンテンツライティング担当
  → Marketing Department/marketing-copywriter

既存: ナレッジ管理担当
  → 横断部署 Knowledge Management Department

当面は物理的に移動しなくてもよい。
まずは agents/README.md や部署INDEXで「どの部署の社員か」を明示するだけでもよい。

7. 田中祐一AIの3つの利用モード

田中祐一AIは、使う人によって権限と出力を分ける。

7-1. Private Mode

田中さん本人用。

読める範囲が最も広い。

  • 原本データ
  • 内部メモ
  • 未整理ナレッジ
  • スタッフ向け資料
  • 受講生には見せない判断
  • 事業戦略

7-2. Staff Mode

スタッフ用。

業務実行に必要な範囲だけを読ませる。

  • 確定済みナレッジ
  • 業務マニュアル
  • 案件情報
  • 出力テンプレート
  • 禁止事項
  • エスカレーション条件

7-3. Student Mode

受講生用。

社内情報を見せず、教材化・抽象化済みのナレッジだけを使う。

  • 公開してよいメソッド
  • 受講生向けテンプレート
  • セルフチェック
  • 例題
  • 添削結果の一般化パターン

この3モードを分けることで、田中祐一AIの価値を外部提供しながら、社内の情報管理も守れる。

8. 精度・独自性・再現性が上がる条件

AI社員を作っただけでは精度は上がらない。

精度が上がる条件は、次の循環があること。

原本
  → ナレッジ候補
  → 田中さん承認
  → knowledgeへ格納
  → AI社員がskillで使用
  → output生成
  → QA
  → 良い判断だけknowledge / SKILLへ還流

独自性は「田中さんの判断」を分解して保存したときに生まれる。

再現性は「毎回同じ工程と確認ゲート」を通すことで生まれる。

したがって、重要なのはエージェント数ではなく、各AI社員に以下が揃っていること。

  • 役割定義
  • 入力
  • 出力
  • 参照ナレッジ
  • 手順
  • 品質基準
  • 保存先
  • QA
  • 改善ログ

9. まず実装する最小セット

最初から全部署を完成させない。

最初の実装はこれでよい。

Marketing Department/
├── cmo/
├── concept-maker/
├── offer-architect/
├── lp-architect/
├── stepmail-writer/
└── marketing-qa-reviewer/

最初に作るskillは、次の5つ。

  1. マーケティング依頼ルーティング
  2. コンセプト構築
  3. オファー構築
  4. LP構造設計
  5. ステップメール設計

その後、必要に応じてシナリオ、広告、SNS、セールス、講座運営へ広げる。

10. 設計上の重要判断

判断1: 田中祐一AIはAI社員を呼ぶべきか

呼ぶべき。

ただし、ユーザーに複雑な選択をさせない。

田中祐一AIが内部で呼び分ける。

判断2: エージェントを細分化すべきか

すべき。

ただし、部署、社員、skillの3層に分ける。

何でも社員にせず、まずskillとして持たせる。

判断3: フォルダは業務部署で括るべきか

括るべき。

今後エージェントが増えるほど、部署単位のINDEXと_RULEが必要になる。

判断4: 田中祐一AIの価値訴求に使えるか

使える。

「田中祐一AIを使える」よりも、「田中祐一のプロモーション設計チームを使える」という訴求にできる。

ただし、受講生に内部構造を全部見せる必要はない。

表側ではシンプルに見せ、裏側で社員群が動く構造がよい。

11. 次に決めること

次に決めるべきは、Marketing Departmentの最初の社員とskillのI/O。

具体的には以下。

  1. cmo は何を入力にして、誰に振り分けるか
  2. concept-maker は何を読んで、何を出すか
  3. offer-architect はコンセプト確定前に動くのか、後に動くのか
  4. lp-architect は本文まで書くのか、構造だけ作るのか
  5. marketing-qa-reviewer はどの段階で止める権限を持つか

この5つが決まると、田中祐一AIとAI社員の関係はかなり明確になる。

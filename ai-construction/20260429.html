# AI運用 全体ガイド（2026-04-29 時点）

このドキュメントは、theleadpromotion のフォルダの中身がどういう構造になっているか、CursorやClaude Codeを使うときに「どこを見ればいいの？」が分かるようにまとめた案内です。

CursorもClaude Codeも触ったことがない人でも分かるように書いていますので、安心して読んでくださいね。

## まずは前提の話

### Cursor と Claude Code って何？

両方とも「AI と一緒に作業ができる文書編集ツール」みたいなものです。
僕（田中）の会社では、人によってどっちを使うかが違うんですよね。

- Cursor を使っている人
- Claude Code を使っている人

このドキュメントは、両方のスタッフが同じ業務をできるようにするための説明書になっています。

### 何が問題だったか

実は2026年4月までは、こんな状態でした。

「あれ？このスキル、Cursor 側にしかないの？」
「議事録の作り方、ファイルが何個もあって、どれが正しいんだろう？」

こういう「同じ業務なのに、ツールごとに違うファイルを見ないといけない」状態だったんですよね。
これだと、ツールが違うだけで、できることが違ってくる。
おかしいですよね。

それで「全員が同じ業務マニュアルを見て動けるようにしよう」と整理し直したわけです。

ルール（書き方の基本ルール）は、CursorとClaude Codeで揃えておかないと、できあがるものが変わってきちゃうので、両方に同じものを置いている、ということですね。

## 「スキル」「ナレッジ」「ルール」って違うの？

このあたり、ちょっとややこしいんですよね。
シンプルに言うとこんな違いがあります。

| | ナレッジ | スキル | ルール |
|---|---|---|---|
| どんなとき使う？ | 「正解は何？」を調べたいとき | 「どうやればいいの？」と手順を確認したいとき | 「いつもこう書こう」という共通ルール |
| 例え | 教科書・参考書 | 業務手順書 | 就業規則 |
| 例 | 「広告のCPA目標は事業者向けで4,000〜8,000円」 | 「広告は3本配信→3日待って→ダメなら入れ替え」 | 「文章で太字を使わない」 |
| 場所 | `/knowledge/` | `/skills/` | `CLAUDE.md` と `.cursor/rules/` |

迷ったらこんなふうに考えてみてください。

- 「これって AI が判断するときに必要な情報？」 → ナレッジ
- 「これって AI が業務をやる順番？」 → スキル
- 「これって毎回守る書き方のルール？」 → ルール

## 実際に使うときの流れ

### Cursor を使っている人

1. Cursorを開くと、`.cursor/rules/` の中のルールが自動で読み込まれます（書き方の共通ルール）
2. スキルを呼び出したいときは、スキルの名前（「Loom取得」とか）を指示します
3. すると `.cursor/skills/Loom取得/` が起動して、本社（`/skills/Loom取得/`）の中身を見て動きます

### Claude Code を使っている人

1. Claude Codeを開くと、`CLAUDE.md` が自動で読み込まれます（書き方の共通ルール）
2. スキルを呼び出したいときは、`.claude/skills/Loom取得/` を起動 → 本社の中身を見て動きます
3. 専門の担当者を呼びたいときは「議事録担当を呼んで」のように指示すると、`.claude/agents/minutes-writer.md` が起動します

### GitHub で見るとき

スタッフが GitHub でファイルを見たいときは、こう見ればいいです。

- 「業務がどう動くか」を知りたい → `/skills/` を見る
- 「判断基準」を知りたい → `/knowledge/` を見る
- 「ルール」を知りたい → `CLAUDE.md` または `.cursor/rules/` を見る
- 「人格カルテ（誰が語るか）」を知りたい → `/configs/<ホルダー名>/` を見る
- 「案件の基本情報（ファネル型・ターゲット・見せ方）」を知りたい → `21_プロジェクト一覧/<案件>/マスターデータ/` を見る

## 編集するときのルール

ここ大事なところです。

### 業務スキル・ナレッジを編集するとき

`/skills/` `/knowledge/` `/agents/` `/configs/` の中だけを直してください。
**`.cursor/skills/` や `.claude/skills/` は触らない**でください。
こちらは「コピー」みたいなものなので、編集しても翌日6:30に上書きされて消えちゃいます。

### ルールを編集するとき

`CLAUDE.md` を直したら、必ず `.cursor/rules/` の対応するファイルも直してください。
逆も同じです。
両方に同じ内容が入っていないと、CursorとClaude Codeで動きが変わっちゃうので。

どのファイルがどのファイルに対応しているかは、`CLAUDE.md` の「ルール索引」を見てください。

### スキル・ナレッジを編集すると、自動で何が起きるか

`/skills/` を編集すると、毎日朝6:30に自動で動くプログラムが、その変更を `.cursor/skills/` と `.claude/skills/` のコピーに反映してくれます。

すぐに反映したいときは、自分で実行してもOKです。

## これから取り組みたい改善のアイデア

これから取り組むといいかな、と思っているものです。

### すぐ着手できそうなもの

- **計画モードを積極的に使う**：複雑な作業のとき、AIに最初に計画を立てさせる機能。やってみたい
- **記憶機能の整備**：やりとりをまたいでAIに覚えてもらう機能

### じっくり検討するもの

- **出力スタイルの切り替え**：タスクごとに「セールス分析モード」「議事録モード」みたいに出力の見た目を切り替える
- **自動学習機能**：「うまくいった」「失敗した」を AI が自動的に学習する仕組み
- **複数AI並列処理**：複数のAIに並列で違う作業をやってもらう機能

### MCPの整理について

MCPっていうのは、AIに外部のサービス（utageとかGoogleカレンダーとか）と連携させる仕組みのことです。
僕が今 utage（宴）や自分のナレッジMCPを繋いでいるんですが、調べた結果、「3個くらいなら問題ない」と分かりました。
ただ、実際にMCPを使う頻度を見直してみて、使っていないものは外してもいいと思います。
特に「自分のナレッジMCP」は、今は `/knowledge/` フォルダがGitHubで参照できる状態なので、もしかしたら役割が重複しているかもしれません。

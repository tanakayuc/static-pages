Claude Code引き継ぎ指示書: 田中祐一AI・AI社員DB化・MCP移行

作成日: 2026-07-03

目的

Claude Codeに、田中祐一AIの根本見直しとAI社員DB化の次フェーズ設計を依頼するための引き継ぎ。

今回の目的は、単にDBテーブルやMCPツールを増やすことではない。

田中祐一AIを「ナレッジ検索AI」から「田中祐一のプロモーション設計チームを動かすAI OS」へ変える。そのために、AI社員、ナレッジ、スキル、ルール、評価基準、利用ログ、育成履歴をDBとMCPでつなぐ。

背景

田中祐一AIはこれまで、主に以下を担っていた。

  1. 田中祐一のナレッジを参照する。
  2. 添削、制作、相談、QA検索を行う。
  3. ナレッジやスキルをMarkdownとして蓄積する。
  4. Cursor / Claude Code / Codex / MCPから利用する。

ただし、運用上の課題が見えてきた。

  1. ナレッジは増えているが、AI社員として役割化されきっていない。
  2. 社長がAIを使いこなしている状態であり、会社としてAI社員を育てている状態になりきっていない。
  3. 過去のグルコン約200件、個人コンサル約70件、Chatwork、Zoom、Loom、教材、LP、PDFなどの資産がMarkdown整理に留まり、評価軸が変わるたびに再仕分けが必要になる。
  4. どのAI社員がどのナレッジを使い、どの成果物を作ったかが追跡できない。
  5. どの作業をスキル化、ナレッジ化、社員化すべきかのフィードバックループがまだ弱い。

今日の方針転換

購入教材「AI社員の作り方」をきっかけに、田中祐一AIの位置づけを見直した。

旧方針:

田中祐一AI = ナレッジを蓄積し、必要な知識を取り出すAI

新方針:

田中祐一AI = AI社員を統括し、プロモーション設計チームを動かすAI OS

マーケティング上の見せ方も変更する。

旧訴求:

田中祐一AIが使える

新訴求:

田中祐一のプロモーション設計チームをAIで使える

ユーザーはAI社員の作り方を知らなくてよい。田中祐一が育てたAI社員群を使えることが価値になる。

AI社員の考え方

AI社員は、単なるプロンプトではない。

AI社員は以下を持つ存在として定義する。

  1. 役割
  2. 担当範囲
  3. 判断基準
  4. 参照スキル
  5. 参照ナレッジ
  6. 成果物
  7. 稼働ログ
  8. 評価履歴
  9. 育成履歴

この9つがDBで追跡できるようになると、「AI社員が働き、評価され、育つ」状態に近づく。

このスレッドでの重要な議論

1. 過去MD整理は無駄ではない

田中さんは、過去2週間から1か月ほどかけて、グルコン、個人コンサル、ナレッジをMD化・仕分けしてきた。

途中で評価軸が変わり、「またゼロからやり直しではないか」という懸念が出た。

今回の結論:

過去MD整理は、最終分類ではなく、旧評価軸による初期ラベル付きデータとして救済する。

つまり、過去作業は消えない。DB上で legacy_md_2026Q2 として比較可能な履歴にする。

2. ナレッジフィードバックループが中核

毎回の作業後に、人間が手動で「スキル化・ナレッジ化・社員化」を完了させるのは重い。

そのため、今後は以下に変える。

作業中に候補をDBへ送る
  → 週次ナレッジレビューで候補を整理
  → 田中祐一が判断
  → 承認されたものだけ rules / skills / knowledge / agents へ反映

ナレッジマネージャーは会社の中核AI社員になる。

3. 田中祐一AIとAI社員は絡める

田中祐一AIとAI社員は別物ではなく、連携させる。

田中祐一AI:

入口、ルーター、統括、最終編集者

AI社員:

専門業務を担当する実行者

例:

田中祐一AI
  → プロモーションディレクター
  → ビジネス設計担当
  → ファネル設計担当
  → コンセプトメイカー
  → オファー設計担当
  → LP設計担当
  → マーケティングQA担当
  → 田中祐一AIが最終編集

4. マーケティング訴求も変える

「田中祐一AIが使える」よりも、「田中祐一のプロモーション設計チームをAIで使える」の方が価値が伝わる。

将来的には、裏側でAI社員がどのように働いているかを可視化し、PRにも使う。

Claude Codeに注意してほしいこと

SQLは勝手に実行しない

田中祐一の運用ルールとして、Supabaseに入れるSQLは必ず田中祐一へ全文提示する。

Claude CodeやCodexが本番DBへ勝手にSQLを適用しない。

knowledge/へ直接反映しない

ナレッジ候補は、まず knowledge_promotion_queue またはナレッジ管理担当へ送る。

承認なしに knowledge/、skills/、agents/、rules/ を直接増やしすぎない。

受講生向け公開に注意

restricted、tanaka_only、staff_only は外部利用に出さない。

初期MCPツールは、admin以外には public / internal までに抑える。

DB化はMarkdown廃止ではない

Markdownは人間が読む正本、DBは索引・履歴・ログ・評価・権限の台帳。

DBにしたからといって、Markdownの正本を捨てない。
